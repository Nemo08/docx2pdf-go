# AGENTS.md

## Стек и Окружение
- Язык: Go (последняя стабильная версия)
- Управление зависимостями: Go modules (`go.mod`)
- Локальный анализатор контекста: 
  - MCP-сервер `gograph` (анализ связей и архитектуры)
  - MCP-сервер `go-dev-mcp` (интеграция с тулчейном Go, запуск тестов, чтение доки)
- ОС: Windows. Системная кодировка консоли: CP866. В выводах командной строки используй ТОЛЬКО английский язык или экранируй кириллицу во избежание кракозябр.

## Команды проекта
- Сборка приложения: `go build -v ./...` (или через инструмент `build` в `go-dev-mcp`)
- Запуск тестов: `go test -v -race -cover ./...` (или через инструмент `test` в `go-dev-mcp`)
- Запуск линтера: `golangci-lint run` (или через инструмент `vet` в `go-dev-mcp`)
- Форматирование: `go fmt ./...` и `goimports -w .` (или автоматический вызов `gofmt` через MCP)
- Предпочтительно используй возможности MCP серверов!!!

## Важные правила автоматизации (CRITICAL)
- **Принцип невмешательства**: Выполняй действия автоматически через терминал/бинарники/MCP без промежуточных вопросов, кроме необратимых деструктивных действий.
- **Работа с контекстом**: 
  - Перед изменением крупных кусков кода ВСЕГДА используй инструмент `gograph` (команда `context <символ>`), чтобы получить зависимости пакета, а не читать файлы вручную.
  - Для быстрого чтения сигнатур функций, документации стандартных библиотек или сторонних пакетов используй инструмент `view_docs` из MCP `go-dev-mcp`.
- **Проверка кода**: После любого изменения кода в проекте автоматически запускай инструмент `test` или `build` из `go-dev-mcp`, чтобы мгновенно локализовать ошибки компиляции, не дожидаясь ручного запуска.
- **Режим планирования**: Перед написанием фичи сначала используй Plan Mode (клавиша <TAB> в OpenCode) для генерации архитектурного плана.
- **Атомарность**: Правки должны быть строго атомарны и по месту. Изменяй только нужные строки, ЗАПРЕЩЕНО переписывать весь файл без необходимости.

## Стандарты кода (Go Style Guide)
- **Архитектура**: Следуй идиоматичной структуре Go (`/cmd` для точек входа).
- **Мьютексы**: При использовании `sync.Mutex` всегда пиши `defer m.Unlock()` сразу после вызова `m.Lock()`.
- **Горутины**: Запрещено запускать горутины (`go func()`) без контроля их жизненного цикла через `sync.WaitGroup` или `context.Context`.
- **Обработка ошибок**: Возвращай ошибки явно через `error`. Не используй `panic` для управления логикой. Ошибки оборачивай через `fmt.Errorf("контекст: %w", err)`.
- **Метрики**: Не усложняй код. Если цикломатическая сложность функции по метрикам `gograph` превышает 10, автоматически разбивай её на мелкие хелперы.
